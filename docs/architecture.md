# Architecture

A single-host Next.js app with SQLite + local-disk uploads. No external services beyond DNS and TLS termination. Hosted on Manus AI.

## Deployment topology

```mermaid
flowchart LR
  subgraph Browser
    Kid[Kid / Admin]
    Recipient[Share-link recipient]
  end

  Kid -- HTTPS + aa_session cookie --> TLS
  Recipient -- HTTPS + share token --> TLS

  TLS[TLS terminator / reverse proxy]

  subgraph ManusHost[Manus AI host - single VM]
    TLS --> Next
    Next[Next.js 15 server<br/>npm start - port 3000]
    subgraph App[App Router]
      SSR[Server components<br/>SSR pages]
      API[Route Handlers<br/>/api/*]
    end
    Next --- SSR
    Next --- API

    SSR -- read --> SQLite[(SQLite<br/>data.db)]
    API -- read/write --> SQLite
    API -- sharp + DOMPurify --> Disk[(public/uploads/<br/>persistent disk)]
    SSR -- static serve --> Disk
  end
```

### Trust boundaries

- Cookie auth (`aa_session`, httpOnly, sameSite=lax, `secure` in prod) gates `/api/*` mutations and owned-gallery views.
- Share tokens (32-char URL-safe) gate `/share/[token]` reads only — no login required, sub-gallery descent via `?g=<id>` is verified by walking the gallery tree.
- Rate limit: 10 failed logins / username / 10 min (recorded in `login_events`).
- SVG uploads are sanitised server-side by DOMPurify before being written to disk; raster thumbnail generated by sharp.

### Persistence

`data.db` (+ `-wal`, `-shm`) and `public/uploads/` **must** live on a persistent volume. The app is **not** serverless-compatible — `better-sqlite3` opens a local file and uploads are written next to the binary.

## Kid workflow

```mermaid
sequenceDiagram
  autonumber
  actor Kid
  participant UI as Browser (Next.js client)
  participant Server as Next.js server
  participant DB as SQLite
  participant FS as public/uploads/

  Kid->>UI: visit /login
  UI->>Server: POST /api/login (username, password)
  Server->>DB: verify hash, insert session, log login_event
  Server-->>UI: Set-Cookie aa_session
  alt must_change_password = 1
    UI->>UI: redirect /change-password?forced=1
    Kid->>UI: submit new password
    UI->>Server: POST /api/me/password
    Server->>DB: update hash, clear flag
  end
  UI->>UI: redirect to own /gallery/:id

  rect rgb(245, 240, 255)
    note over Kid,FS: Upload art
    Kid->>UI: drag/drop image, PDF, or SVG
    UI->>Server: POST /api/upload (multipart)
    Server->>Server: sharp - thumb + EXIF rotate<br/>DOMPurify - sanitise SVG
    Server->>FS: write file + thumb
    Server->>DB: insert artwork, bump storage_bytes
    Server-->>UI: refresh gallery
  end

  rect rgb(240, 250, 245)
    note over Kid,DB: Share a gallery
    Kid->>UI: open Shares panel, set recipient + expiry
    UI->>Server: POST /api/galleries/:id/shares
    Server->>DB: insert share with 32-char token
    Server-->>UI: return share URL with token
  end

  Kid->>UI: visit /explore
  UI->>Server: SSR list of other kids' top-level galleries
  Kid->>UI: open another kid's gallery (read-only, watermarked)
```

Key paths:
- `/gallery/:id` — owned (edit) vs. read-only depending on `gallery.user_id === user.id`.
- `/explore` — browse other kids' top-level galleries.
- `/change-password` — voluntary or forced (`?forced=1`).

## Admin workflow

```mermaid
sequenceDiagram
  autonumber
  actor Admin
  participant UI as Browser
  participant Server as Next.js server
  participant DB as SQLite

  Admin->>UI: visit /login
  UI->>Server: POST /api/login
  Server->>DB: verify, insert session, log login_event
  Server-->>UI: Set-Cookie aa_session
  UI->>UI: redirect /admin (admin layout guard)

  rect rgb(255, 245, 240)
    note over Admin,DB: Reset a kid's password
    Admin->>UI: /admin/users -> Reset
    UI->>Server: POST /api/admin/users/:id/reset-password
    Server->>DB: hash new pw, set must_change_password=1<br/>insert admin_actions row
    Server-->>UI: show one-time password
    note right of Admin: Capture pw,<br/>hand to kid out-of-band
  end

  rect rgb(240, 245, 255)
    note over Admin,DB: Audit & oversight
    Admin->>UI: /admin/logins
    UI->>Server: SSR last N login_events
    Admin->>UI: /admin/shares
    UI->>Server: SSR all shares + view counts
    Admin->>UI: revoke share
    UI->>Server: DELETE /api/shares/:id
    Server->>DB: set revoked_at, log admin_actions
    Admin->>UI: /admin/audit
    UI->>Server: SSR admin_actions log
  end
```

Admin has no gallery of their own; admin pages are guarded by `app/admin/layout.tsx` (`role === 'admin'`).

## Share-recipient workflow

```mermaid
sequenceDiagram
  autonumber
  actor Recipient as Recipient (no login)
  participant UI as Browser
  participant Server as Next.js server
  participant DB as SQLite
  participant FS as public/uploads/

  Recipient->>UI: visit /share/TOKEN (optionally ?g=N)
  UI->>Server: SSR share lookup
  Server->>DB: find share by token
  alt revoked or expired
    Server-->>UI: 404
  else valid
    Server->>Server: if ?g=N, verify isDescendant(N, share.gallery_id)
    Server->>DB: insert share_views (IP, UA)
    Server->>DB: list sub-galleries + artworks
    Server-->>UI: render read-only gallery<br/>with owner-name watermark
  end
  Recipient->>UI: click thumbnail
  UI->>UI: open lightbox (zoom/rotate/pan;<br/>NoDownloadGuard blocks right-click + drag)
  UI->>FS: GET /uploads/FILE (full resolution)
```

Notes on recipient experience:
- No account, no cookie. Token is the only credential.
- `NoDownloadGuard` is a soft mitigation only (right-click off, drag off, watermark, PDF toolbar hidden) — **not DRM**. Screenshots remain possible.
- Sub-gallery breadcrumb stays inside the shared subtree; descending out of it returns 404.
